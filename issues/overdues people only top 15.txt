The Core Problem
Your current query_router.py is a hand-written intent classifier. Every time a user phrases something slightly differently, it fails and needs a patch. This doesn't scale. You're essentially doing what a model should do.

Your Options

Replace the entire regex system with a single Groq call using llama-3.1-8b-instant in JSON mode:

Pros: Handles any phrasing, zero maintenance, already in your stack, very fast on Groq, adds params extraction for free (customer name, limit, date range)

Cons: Adds ~80-150ms latency on every request, uses ~50-100 tokens per query even for simple DB lookups, needs a well-written system prompt

Best fit for your situation. Token cost is negligible — Groq's 8b model pricing makes 1 million routing calls cost under $1.

Option B — Semantic Router (ChromaDB embeddings, already in stack)
Store 10-15 example phrases per intent in ChromaDB. For each query, embed it and find the closest intent by cosine similarity.

python
# "overdues people only top 15" → embedding → nearest → "overdue_list"

Cons: Needs example phrases per intent (~200 examples total), embedding overhead (~20-40ms), harder to extract params (still need regex for limit/customer), cold start if ChromaDB is empty

Option C — Keep regex, make it smarter (current path)
Add more patterns, better exemptions, unit tests that catch regressions.

Pros: Zero latency, zero tokens, works offline

Cons: You're already seeing the problem — every edge case needs a new rule. Unmaintainable at scale. You'll be doing this forever.

Option D — Ditch pre-routing entirely, use tool-calling
Remove the router. Every query goes to the LLM with tools (get_overdue_list, get_client_summary, etc.). The model decides which tool to call.

Pros: Most natural, handles complex multi-intent queries, already partially there with AI_SIMPLE tool-calling

Sharp catch. You've identified the two real problems with naive LLM routing:

Problem 1: LLM doesn't know your customers
"do you know nilgiris fresh?" — the LLM can extract the text "nilgiris fresh" but has no idea if it's a valid customer. It might hallucinate the name, get the casing wrong, or return "Nilgiris Fresh Ltd" instead.

The fix is already in your code. The LLM only extracts a hint ("nilgiris fresh"), and _extract_customer_name() in direct_query_handler.py queries the DB for all your actual customer names and fuzzy-matches. The DB is always the source of truth — not the LLM. The LLM just says "this looks like a customer query, extracted hint = nilgiris fresh". The DB resolves it to the exact name.

Problem 2: Few examples — how does it generalize?
This is where the real architectural split matters. There are two completely different things the router needs to do:

"do you know nilgiris fresh?" 
       │
       ├─ INTENT TYPE: "client lookup" ← LLM handles this well with few examples
       │                                  (it understands "do you know X" = info request)
       │
       └─ ENTITY: "nilgiris fresh"     ← DB handles this, not LLM
The LLM only needs to classify intent type — which is a conceptual pattern, not domain knowledge. It doesn't need to know your business data. It generalizes: "tell me about X", "do you know X", "what's up with X" → all mean "client_summary". That's language understanding, not business knowledge.

The Better Architecture: Embeddings (already in your stack)
Since you already have ChromaDB running, the cleanest solution is semantic routing with embeddings — zero LLM tokens, offline, improves with examples:

Free text ──► embed (sentence-transformers tiny model)
                      │
                      ▼
            ChromaDB intent collection
            (10-15 examples per intent)
                      │
                      ▼
            nearest match + confidence score
                      │
              ┌───────┴────────┐
           high conf         low conf
              │                  │
         intent_key           AI_SIMPLE
              │              AI_COMPLEX
              ▼
          intents.py (unchanged)
Store examples like:

"overdue_list":  ["show overdues", "who hasn't paid", "top 15 overdue people", "overdues customers only"]
"client_summary": ["tell me about X", "do you know X", "what's X's status"]
"ai_complex":    ["analyse my cash flow", "why is my collection rate low", "give me a plan"]
Pros: Zero tokens, ~15ms, handles paraphrase naturally, works offline, improves just by adding examples — no code changes
Cons: Needs a small embedding model (~80MB, one-time download), needs 10-15 seed examples per intent
